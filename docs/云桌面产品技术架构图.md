# 自研云桌面（VDI）全栈技术架构选型
---

## 1. 选型总览

### 1.1 组件选型矩阵

| 分层 | 选型 | 协议 | 用途 |
|:--|:--|:--|:--|
| 编排 | Kubernetes + KubeVirt | Apache-2.0 | 统一调度 Linux 容器与 Windows VM |
| 计算 | KubeVirt (KVM) | Apache-2.0 | Windows 完整 VM，GPU 直通 |
| 容器运行时 | Custom runc + Containerd | Apache-2.0 | Linux 容器桌面，OverlayFS 劫持 |
| 存储 | Longhorn | Apache-2.0 | 分布式块存储，3 副本强一致 |
| 网络 | Kube-OVN | Apache-2.0 | OVS 软件定义网络，VLAN 桥接 |
| 监控 | Prometheus Operator + Grafana | Apache-2.0 | 指标采集 + 告警自愈 |
| 信令穿透 | Coturn TURN Server | BSD-3-Clause | STUN / TURN / NAT 穿透 |
| 媒体 | Pion WebRTC / webrtc-rs | MIT | WebRTC 媒体网关 |
| 编解码 | GStreamer (x264 / nvh264 / VAAPI) | LGPL-2.1 | 屏幕捕获 + 硬编 + 推流 |
| 客户端 | Tauri (Rust) | MIT / Apache-2.0 | 原生客户端外壳 |

## 2. 七层架构

```
┌──────────────────────────────────────────────────────┐
│ L1  客户端层       Web 客户端 · Tauri Native 客户端    │
├──────────────────────────────────────────────────────┤
│ L2  媒体与接入层   Coturn · Pion WebRTC · GStreamer   │
├──────────────────────────────────────────────────────┤
│ L3  业务编排层     云桌面控制台 · Session Broker       │
├──────────────────────────────────────────────────────┤
│ L4  云桌面交付层   Linux 容器 Pod · Windows VM         │
├──────────────────────────────────────────────────────┤
│ L5  K8s 工作负载   KubeVirt · Longhorn · Kube-OVN     │
│                    · Prometheus Operator              │
├──────────────────────────────────────────────────────┤
│ L6  K8s 编排层     Kubernetes 控制面                  │
├──────────────────────────────────────────────────────┤
│ L7  基础设施层     物理服务器 · 物理网络 · 物理存储     │
└──────────────────────────────────────────────────────┘
```

- **L7 → L6 自下而上支撑**：物理硬件承载 K8s 控制面
- **L5 是 K8s 内的工作负载**：四大件（计算/存储/网络/监控）以 Operator + 守护进程方式运行
- **L4 是用户实际拿到的实例**：Linux 容器与 Windows VM
- **L3 是业务控制面**：把 K8s 能力包装成"云桌面控制台"，对租户提供 VM / Pod 自服务
- **L2 / L1 是端到端媒体与交互**：WebRTC 推流 + 客户端渲染

## 3. 关键选型理由

### 3.1 为什么用 Kubernetes + KubeVirt

- 统一调度平面：Linux 容器与 Windows VM 都被包装为 K8s Pod，API 声明式一致
- KubeVirt 成熟稳定：直接复用 K8s 调度、扩缩、监控、权限体系
- GPU 直通 / 嵌套虚拟化：通过 KubeVirt CRD 声明即可，host-passthrough 模式

### 3.2 为什么用 Longhorn

- 块存储而非对象存储：满足 Windows 系统盘 + 高 IOPS 编译场景
- 3 副本强一致：节点故障秒级恢复，数据零丢失
- 数据本地化读：VDI Pod 同节点保留一份副本，读延迟接近物理机

### 3.3 为什么用 Kube-OVN

- 基于 OVS 的成熟 SDN：万兆性能可预期
- VLAN 桥接企业内网：云桌面开机即获内网 IP，免 NAT 映射
- 多租户二层隔离：Subnet + ACL 原生支持外包场景

### 3.4 为什么用 Pion WebRTC

- Go 原生实现：与 K8s 控制器、Broker 同语言，技术栈统一
- MIT 协议：可深度定制并闭源
- 媒体 + DataChannel 双通道：屏幕流 + 键鼠/USB 控制复用一条连接

### 3.5 为什么用 GStreamer

- 全平台硬编支持：NVIDIA (nvh264enc) / Intel (VAAPI) / 软件 (x264) 一套管线
- LGPL 协议合规：进程隔离 + 动态链接，闭源产品可放心使用
- 生态成熟：屏幕捕获、推流、水印叠加等场景均有现成插件

### 3.6 为什么用 Tauri

- 极致轻量：包体积 < 15 MB
- Rust 后端：可写系统级 Hook、URB 拦截等特权功能
- MIT / Apache-2.0：闭源分发无障碍

## 4. 核心能力支撑

| 业务诉求 | 技术方案 |
|:--|:--|
| 用户 `sudo apt-get` 安装任意软件 | Custom runc + OverlayFS 外部挂载劫持 |
| 容器重启不丢数据 | upperdir 落在 Longhorn PVC，K8s 重建 Pod 自动挂回 |
| 早上 9 点开机风暴 | Longhorn 3 副本 + 数据本地化读 |
| 物理多显示器扩展 | IDD 间接显示驱动 + 多轨 PTS 同步推流 |
| 抗屏摄泄密追溯 | DWT-DCT 频域盲水印 + 透视校正解码 |
| Web 端零安装 | HTML5 Canvas 客户端，DataChannel 传输键鼠 |
| 工位重度生产力 | Tauri Native + Rust 键盘 Hook + USBIP 重定向 |
| 死锁自动恢复 | Prometheus 指标 → Alertmanager → 自研 Broker 自愈闭环 |
